# 第一阶段 PoC 计划

## 目标

验证一个文本测试用例可以通过 AI 辅助探索，转换成稳定的 Appium 自动化脚本。

本阶段输出不是完整框架，而是一个经过验证的最小闭环。

在评价 AI 生成能力前，本阶段先增加一个人工稳定基线前置验证步骤。该步骤仍属于第一阶段 PoC，不升级为独立 Phase 0，也不要求人工手写全部代码。它用于先证明 adb、Appium、目标 App、测试场景、locator、等待和断言本身可以稳定工作，再用同一个用例验证 AI 生成流程。

## 范围

范围内：

- Android 真机或模拟器通过 adb 连接。
- Appium Server 基础配置。
- Appium MCP 基础配置，用于 AI 辅助探索手机页面。
- 一个目标 App。
- 一到三个简单测试用例。
- Python + pytest + Appium 生成脚本。
- 失败时基础诊断产物收集。
- 最小日志能力，包括运行日志、失败截图、失败 page source 和基础 logcat。

范围外：

- 大规模批量生成。
- 完整页面对象库。
- iOS 支持。
- 多设备兼容矩阵。
- 云真机执行。
- 每次回归都由 AI 自主执行和判断。

## 目标流程

```text
人工确认 CASE00 测试意图、步骤和预期结果
  -> 使用 Codex 辅助实现确定性 Appium 基线脚本
  -> 验证 locator、显式等待和断言
  -> 连续重复执行并记录成功率和失败原因
  -> 建立稳定参考版本
  -> 输入同一个文本测试用例
  -> AI 读取前置条件、步骤和预期结果
  -> AI 通过 Appium MCP 检查和操作手机
  -> AI 使用截图、page source、adb、日志纠正探索错误
  -> AI 用证据确认预期结果
  -> AI 生成 pytest + Appium 代码
  -> 与人工确认基线比较
  -> AI 生成脚本至少执行 1 次验证
  -> 失败时保存诊断产物
```

## AI 生成前的人工稳定基线

### 目的

人工稳定基线用于在评价 AI 生成能力前，先隔离和验证基础环境与测试场景，避免 AI 生成脚本失败时无法判断问题来源。

需要先排除或定位的问题包括：

- adb 或手机连接不稳定。
- Appium Server 无法正常建立 session。
- Appium Driver 或 capabilities 配置不正确。
- 目标 App 启动、登录态或页面状态不稳定。
- locator 不稳定或只能依赖坐标。
- 页面加载、动画和等待条件处理不充分。
- 测试数据或设备在线状态不满足前置条件。
- 测试场景本身不可重复。
- AI 理解测试意图或生成代码错误。

### 执行流程

```text
人工确认 CASE00 测试意图、步骤和预期结果
  -> 使用 Codex 辅助实现确定性 Appium 基线脚本
  -> 验证 locator、显式等待和断言
  -> 连续重复执行
  -> 记录成功率和失败原因
  -> 建立稳定参考版本
  -> 再使用相同 CASE00 验证自然语言、Appium MCP 和 AI 脚本生成流程
  -> 比较 AI 生成版本和人工确认基线
```

### 基线验收标准

人工稳定基线满足以下条件后，才作为后续 AI 生成实验的参考版本：

1. `adb devices` 可以稳定识别目标 Android 设备。
2. Appium Server 可以正常建立 session。
3. 可以启动或连接目标 App。
4. CASE00 可以完整执行。
5. 脚本使用稳定 locator。
6. 优先使用显式等待，避免无意义固定 `sleep`。
7. 对关键预期结果有明确断言。
8. 测试结束后不破坏下一次运行所需的前置条件。
9. 不需要人工重新配网或恢复设备绑定。
10. 同一设备、同一 App 版本、同一测试数据下至少成功执行 1 次。
11. 失败时可以保留截图、page source、运行日志和失败步骤上下文。

### CASE00 定位

当前第一条人工稳定基线候选为 `appium_auto/specs/raw/CASE00-进入设备详情页并查看设备信息.txt`。

选择原因：

- 用例目标是进入已绑定设备的详情页和高级设置页进行查看。
- 用例结束要求返回设备详情页和首页。
- 预期结束状态仍然保留设备 `V4#4`，不修改设备绑定关系。
- 不需要人工重新配网，适合连续重复执行验证。
- 预期结果包含设备详情页信息、温度区域、湿度区域和高级设置页“移除设备”按钮，具备可观察断言。

该用例当前定位为“稳定基线候选”，不是已经验证通过的稳定基线。只有在满足基线验收标准，尤其是成功执行 1 次后，才能把它作为后续 AI 生成版本的参考脚本。

测试步骤和预期结果以用例文件为准，规划文档不重复维护完整正文。

### CASE01 定位

`appium_auto/specs/raw/CASE01删除设备成功.txt` 保留为后续破坏性测试、人工准备测试或特殊回归用例，不作为第一条人工稳定基线。

原因：

- 用例成功后会解除设备绑定。
- 下一次执行前需要人工重新配网或恢复设备绑定。
- 不适合连续重复执行稳定性验证。
- 该用例仍有业务价值，但应在明确环境恢复方式后再进入自动化验证。
- 自动化脚本默认不参与普通回归，只有显式设置 `RUN_DESTRUCTIVE_CASE01=1` 时才允许执行；验证成功后立即停止，不重复运行。

### CASE02 定位

`appium_auto/specs/raw/CASE02删除设备后重新配网成功.txt` 用于验证完整的“删除设备 -> 三分钟内重新配网 -> 恢复设备”闭环。

- 用例执行期间会解除设备绑定，但成功后会恢复设备联网状态，因此属于可重复的有状态测试，不属于破坏性测试。
- 配网成功后应恢复名称为 `V4#4` 的在线设备，理论上可以作为后续循环统计配网成功率的基础。
- 使用 `RUN_STATEFUL_CASE02=1` 显式执行，不进入普通 pytest 默认执行。
- 如果配网失败或超过设备的三分钟配网窗口，应停止执行并保留诊断产物，由人工恢复设备配网状态。
- CASE02 只使用 App 已默认填充的当前 Wi-Fi 信息；独立 Wi-Fi 配置预留给后续 CASE03 使用。

## 公共能力和环境配置

CASE00、CASE01、CASE02 验证完成后，进入最小 Phase 2 框架化提取：

- 环境配置采用 `default.yaml` 与本机 `local.yaml` 分层合并，运行时环境变量优先级最高。
- 真实 Wi-Fi SSID 和密码单独保存在 Git Ignore 的 `wifi.local.yaml`，CASE00-02 暂不读取。
- pytest 统一管理 environment、driver 和运行上下文。
- 页面 locator 和页面操作沉淀到 Page Object，跨页面删除和配网流程沉淀到设备生命周期 Flow。
- 测试脚本只保留业务步骤和断言，不重复实现 driver、等待、日志和诊断工具。

## AI 生成版本对比指标

人工稳定基线通过后，使用同一个 CASE00 进行 AI 生成实验。当前阶段只定义比较方法，不建设完整 AI Eval 平台。

至少比较：

- 是否完整保留原始测试意图。
- 是否遗漏或擅自增加步骤。
- locator 是否稳定。
- 是否使用不必要的坐标或脆弱 XPath。
- 是否使用合理的显式等待。
- 是否有完整断言。
- 是否可以独立于 AI 重复执行。
- 连续执行成功率。
- 生成后需要人工修改多少次。
- AI 生成版本与人工确认基线之间的主要差异。

## 本地能力要求

- `adb devices` 可以识别目标手机或模拟器。
- Appium Server 可以启动并创建 session。
- 目标 App 的 package/activity 已知，或可以被发现。
- Appium MCP 可以连接到 Appium 环境。
- Python 环境可以运行 pytest 和 Appium Python client。

pytest 用例通过项目级 session fixture 获取 Appium Server：执行前先检查配置地址的 `/status`；本机服务不可用时由 fixture 启动 Appium Server，并等待服务就绪后再创建 driver。执行结束时只关闭本次 fixture 自己启动的进程；如果服务原本由开发者或其他工具启动，则复用且不关闭，避免干扰共享调试环境。

## 第一条测试用例要求

第一条用例应尽量简单、稳定。

第一条人工稳定基线候选已选定为 `appium_auto/specs/raw/CASE00-进入设备详情页并查看设备信息.txt`。后续如更换第一条用例，应继续满足“非破坏性、可重复执行、可观察断言、执行后不需要人工恢复环境”的要求。

适合作为第一条的候选：

- 启动 App 并验证首页出现。
- 使用固定测试账号登录。
- 进入稳定的设置页或关于页。
- 验证已知静态文案或页面标题。

不适合作为第一条的候选：

- 复杂滚动列表操作。
- 动态后端数据检查。
- 支付、绑定、OTA 或不可逆操作。
- 对网络稳定性依赖很重的流程。
- 需要多个外部设备配合的流程。

## 生成脚本质量要求

生成脚本应满足：

- 使用显式等待。
- 除非没有更好的状态信号，否则避免固定 sleep。
- 优先使用稳定 locator。
- 对预期结果有明确断言。
- 失败时收集截图和 page source。
- 失败时保存运行日志和基础 logcat。
- 不依赖 AI 即可执行。
- 代码可读，便于人工 review。

## 验收标准

第一阶段满足以下条件时视为通过：

1. 设备可以通过 adb 识别。
2. Appium 可以启动或连接目标 App。
3. CASE00 人工稳定基线脚本可以完整执行。
4. 人工稳定基线在同一设备、同一 App 版本、同一测试数据下至少成功执行 1 次。
5. AI 可以通过 Appium MCP 检查当前页面。
6. AI 可以通过探索完成同一个 CASE00 文本测试用例。
7. 为该用例生成 pytest + Appium 脚本。
8. AI 生成脚本可以与人工稳定基线进行质量比较。
9. AI 生成脚本在同一设备和同一 App 版本上至少成功执行 1 次。
10. 构造一次失败时，能产出有用诊断信息：
   - 截图。
   - page source。
   - 相关日志文件或 logcat 片段。
   - 失败步骤或断言上下文。
11. 每次执行有独立 run 目录或等效索引，可以把截图、page source、日志和报告关联起来。

## 风险和控制

### AI 找到的路径难以脚本化

控制方式：

- 探索阶段优先选择可以用稳定 locator 表达的交互。
- 拒绝坐标点击作为正式方案，除非明确标注为临时手段。

### 脚本成功一次但无法稳定重复

控制方式：

- 要求成功执行 1 次。
- 用显式等待替换固定 sleep。
- App 状态泄漏时增加 setup/reset 步骤。

### 预期结果过于模糊

控制方式：

- 把预期结果转换成可观察断言。
- 只有当预期无法从 UI、日志或设备状态观察时，才要求补充说明。
