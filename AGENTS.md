# 项目协作约束

本文件是 Codex 在本项目中的仓库级指导文件。新开对话或重新进入本项目时，应先阅读本文件，再阅读相关阶段文档。

## 默认阅读顺序

1. `AGENTS.md`
2. `doc/plan.md`
3. 当前阶段文档，例如 `doc/phase_01_poc/plan.md`

如果用户修改了文档，以最新文档内容为准。

## 当前阶段

当前默认阶段是第一阶段 PoC：`doc/phase_01_poc/plan.md`。

除非用户明确要求，不要提前实现第二阶段或第三阶段的复杂能力。

## 语言要求

面向人阅读的内容尽量使用中文，包括：

- plan 文档。
- README。
- 代码注释。
- 日志文本。
- 命令行输出。
- 错误提示。
- AI 提示语。
- 测试用例描述。
- 测试报告。

Appium、Appium MCP、Appium Server、Appium Inspector、pytest、adb、Android、iOS、YAML、JSON、XPath、accessibility id 等专有名词可以保留英文。

如果中文表达会明显降低准确性，可以保留英文，并在第一次出现时补充中文解释。

代码注释默认使用中文，只在逻辑不直观、存在取舍、或需要解释业务背景时添加。不要写重复代码含义的注释。

日志默认使用中文，说明当前动作、关键参数和失败原因。错误提示要给出可操作信息，例如缺少哪个环境变量、哪个命令失败、下一步建议检查什么。

## 项目定位

本项目是 AI 辅助 Appium 自动化脚本生产框架。

AI 可以用于：

- 根据文本测试用例探索手机页面。
- 辅助生成 Appium 脚本。
- 辅助排查脚本失败。
- 辅助维护 locator、等待条件和断言。

正式回归执行阶段不应依赖 AI 每一步推理。生成后的脚本应可以独立、稳定、重复执行。

## 实现原则

- 修改方案时先更新文档，再改实现。
- 第一阶段优先验证完整闭环，不提前做大型框架抽象。
- 生成脚本优先使用稳定 locator 和显式等待。
- 避免把坐标点击作为正式脚本方案。
- 每个测试用例都要有明确断言。
- 失败时要保留截图、page source、日志和失败步骤上下文。
- 代码注释默认使用中文，只在必要时添加。

## 测试用例质量门禁

当用户提供自然语言、TXT、Markdown 或其他非结构化测试用例，并要求生成 Appium/pytest 自动化脚本、使用 Appium MCP 探索、批量生成自动化或将手工用例自动化时，必须先使用项目级 Skill：`.agents/skills/testcase-to-yaml/`。

只有同时满足以下条件，才允许进入 Appium 探索和代码生成：

```yaml
quality_gate:
  gate_status: "passed"
  automation_ready: true
```

如果状态为 `review_required` 或 `blocked`，暂停后续自动化流程，输出质量问题并提示用户补充必要信息。不要因为用户直接要求生成代码而绕过质量门禁。

## 工作方式

- 开始实现前，先确认当前任务属于哪个阶段。
- 如果用户给出新的文本测试用例，先通过 `testcase-to-yaml` 进行结构化和质量门禁；通过后按用户要求沉淀到 `appium_auto/specs/`，再进入探索或脚本生成。
- 如果发现文档和实现冲突，先指出冲突，再按用户确认或最新文档继续。
- 不为了“完整”增加暂时没有使用场景的模块。
